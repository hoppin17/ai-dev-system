# Workflow Documents

이 폴더는 AI와 함께 프로젝트를 진행할 때 사용하는 workflow 문서 템플릿을 관리한다.

workflow 문서는 프로젝트의 맥락, 현재 작업, 다음 작업, 기술 구조, 의사결정을 분리해서 기록하기 위한 문서다.
문서는 프로젝트 단위와 서비스 단위에서 다르게 사용될 수 있으며, 어느 단위에서 사용하는지에 따라 작성 범위가 달라진다.

---

## 기본 원칙

- 문서는 코드를 대신하지 않는다.
- 문서는 작업 범위와 판단 기준을 분명히 하기 위해 사용한다.
- 한 문서에 모든 내용을 담지 않는다.
- 각 문서는 하나의 역할만 담당한다.
- 작은 서비스는 최소 문서만 사용하고, 복잡해질 때 선택 문서를 추가한다.

---

## 사용 단위

workflow 문서는 프로젝트 단위와 서비스 단위에서 다르게 사용한다.

이 저장소에서 프로젝트는 여러 서비스를 묶는 상위 단위이고, 서비스는 실제로 구현되는 개별 단위다.

프로젝트 단위에서는 전체 방향, 공통 결정, 구조, 서비스 목록을 관리한다.
서비스 단위에서는 해당 서비스의 목적, 현재 상태, 작업 범위, 다음 작업을 관리한다.

`PROJECT_CONTEXT`는 이름은 유지하되, 프로젝트 단위에서는 전체 프로젝트의 context 문서로 사용하고, 서비스 단위에서는 해당 서비스의 context 문서로 사용할 수 있다.
서비스 단위 context가 프로젝트 단위 context와 충분히 달라질 필요가 생기면, 이후 `SERVICE_CONTEXT` 문서로 분리한다.

---

## 프로젝트 단위 문서

프로젝트 단위에서는 아래 문서를 기본으로 관리한다.

1. `PROJECT_CONTEXT`
2. `SERVICE_OVERVIEW`
3. `ARCHITECTURE`
4. `CURRENT_TASK`
5. `NEXT_TASK`
6. `DECISION_LOG`

프로젝트 단위의 `PROJECT_CONTEXT`는 전체 프로젝트의 목적, 방향, 현재 상태를 정리한다.
`DECISION_LOG`는 서비스별 작은 메모가 아니라, 프로젝트 전체에 영향을 주는 중요한 결정을 기록한다.

---

## 서비스 단위 문서

서비스 단위에서는 아래 문서를 우선 만든다.

1. `PROJECT_CONTEXT`
2. `SERVICE_OVERVIEW`

`PROJECT_CONTEXT`는 해당 서비스의 목적, 현재 상태, 진행 방향을 정리한다.
`SERVICE_OVERVIEW`는 해당 서비스가 무엇을 제공하는지 빠르게 이해하기 위한 개요를 정리한다.

작은 서비스나 초기 단계에서는 `CURRENT_TASK`와 `NEXT_TASK`를 생략하고, `PROJECT_CONTEXT`만 갱신하면서 진행할 수 있다.

작업 범위가 커지거나 여러 단계로 나뉘면 `CURRENT_TASK`와 `NEXT_TASK`를 추가한다.
기술 구조가 복잡해지면 `ARCHITECTURE`를 추가한다.

서비스 단위에서는 보통 `DECISION_LOG`를 따로 만들지 않는다.
프로젝트 전체 방향이나 구조에 영향을 주는 결정은 프로젝트 단위 `DECISION_LOG`에 기록한다.

---

## 문서 역할

각 workflow 문서는 프로젝트 단위와 서비스 단위에서 다르게 사용될 수 있다.
문서 이름은 같더라도, 어느 단위에서 사용하는지에 따라 적는 내용의 범위가 달라진다.

---

### PROJECT_CONTEXT

프로젝트 또는 서비스의 목적, 현재 상황, 방향을 정리하는 문서다.

프로젝트 단위에서:

- 전체 프로젝트의 목적
- 여러 서비스를 묶는 큰 방향
- 전체 진행 상태
- 공통 문제와 리스크
- 아직 결정되지 않은 주요 사항
- 프로젝트 전체 우선순위

서비스 단위에서:

- 해당 서비스의 목적
- 서비스의 현재 상태
- 서비스 단위의 주요 문제
- 서비스 진행 방향
- 해당 서비스에서 우선할 것
- 해당 서비스에서 아직 정하지 않은 것

적지 않는 내용:

- 세부 구현 작업 목록
- 코드 구조 상세
- 데이터 흐름 상세
- 일회성 작업 기록

`PROJECT_CONTEXT`는 “현재 무엇을 위해 진행 중이고, 어디로 가려는가”를 이해하기 위한 문서다.
이름은 유지하되, 서비스 단위에서는 해당 서비스의 context 문서로 사용할 수 있다.

---

### SERVICE_OVERVIEW

실 사용자가 서비스를 빠르게 이해하기 위한 개요 문서다.
개발자는 물론 비개발자도 이해할 수 있도록 서비스가 해결하는 문제와 사용 흐름을 중심으로 정리한다.

프로젝트 단위에서:

- 프로젝트가 해결하려는 문제
- 프로젝트 안에 포함된 서비스 목록
- 각 서비스의 역할 요약
- 서비스 간 관계

서비스 단위에서:

- 해당 서비스의 목적
- 해결하는 문제
- 핵심 기능
- 입력과 출력
- 사용 흐름
- 현재 제공 범위

적지 않는 내용:

- 내부 구현 세부
- 개발 작업 목록
- 기술 부채
- 의사결정 과정

`SERVICE_OVERVIEW`는 서비스를 처음 보는 사람이 빠르게 이해할 수 있도록 돕는 문서다.
프로젝트 단위에서는 서비스 목록과 관계를 설명하고, 서비스 단위에서는 개별 서비스의 개요를 설명한다.

---

### CURRENT_TASK

현재 진행 중인 하나의 작업 범위를 정리하는 문서다.

프로젝트 단위에서:

- 현재 집중하고 있는 큰 작업 단위
- 새로 만들거나 정리 중인 서비스
- 여러 서비스에 걸친 공통 작업
- 공통 문서 구조 변경
- 공통 규칙 변경
- 서비스 단위로 나누기 전의 작업 범위

서비스 단위에서:

- 해당 서비스에서 현재 진행 중인 작업
- 이번 작업에 포함되는 범위
- 이번 작업에서 제외할 항목
- 수정 대상
- 예상 완료 기준
- 확인할 검증 항목

적지 않는 내용:

- 프로젝트 전체의 장기 계획
- 다음 작업 후보 목록
- 완료된 작업의 긴 기록
- 현재 작업과 무관한 아이디어

`CURRENT_TASK`는 지금 실행할 작업을 흔들리지 않게 고정하기 위한 문서다.
작업 중 범위가 커지면 바로 포함하지 말고, `NEXT_TASK`나 `DECISION_LOG`로 분리한다.

---

### NEXT_TASK

앞으로 `CURRENT_TASK`로 내려서 처리할 작업 후보를 정리하는 문서다.

공통으로 적는 내용:

- 다음 작업 후보
- 보류된 작업
- 확인이 필요한 사항
- 우선순위

프로젝트 단위에서 사용할 때는 여러 서비스에 걸친 개선 후보나 서비스 단위로 나누기 전의 큰 작업 후보를 함께 정리할 수 있다.
서비스 단위에서 사용할 때는 해당 서비스 안에서 다음에 처리할 작업 후보를 정리한다.

적지 않는 내용:

- 현재 작업의 세부 구현 내용
- 프로젝트 또는 서비스의 긴 배경 설명
- 이미 완료된 작업의 상세 기록

`NEXT_TASK`는 앞으로 처리할 작업 후보를 관리하는 문서다.
다음 작업으로 전환할 수 있도록 작업 후보, 보류 항목, 확인이 필요한 사항을 함께 정리한다.

---

### DECISION_LOG

중요한 의사결정을 기록하는 문서다.

프로젝트 단위에서:

- 프로젝트 전체 방향에 영향을 주는 결정
- 여러 서비스에 공통으로 적용되는 규칙
- 문서 구조 변경
- 기술 선택
- 아키텍처 방향
- 되돌리기 어려운 결정

서비스 단위에서:

- 기본적으로 별도 작성하지 않는다.
- 단, 해당 서비스 안에서만 중요한 기술 결정이 반복되거나, 서비스 구조가 충분히 복잡해진 경우에는 별도 작성할 수 있다.

적지 않는 내용:

- 단순 작업 메모
- 결정 없이 지나간 생각
- 이미 다른 문서에 충분히 설명된 내용
- 사소한 구현 선택

`DECISION_LOG`는 “왜 이렇게 했는가”를 남기기 위한 문서다.
작은 서비스 단위의 판단은 보통 `PROJECT_CONTEXT`나 `CURRENT_TASK`에 남기고, 프로젝트 전체에 영향을 주는 결정은 프로젝트 단위 `DECISION_LOG`에 기록한다.

---

### ARCHITECTURE

프로젝트 또는 서비스의 기술 구조와 흐름을 정리하는 선택 문서다.

프로젝트 단위에서:

- 전체 시스템 구성
- 서비스 간 관계
- 공통 모듈
- 전체 데이터 흐름
- 외부 시스템과의 연결
- 공통 기술 제약

서비스 단위에서:

- 해당 서비스의 내부 구조
- 주요 모듈과 역할
- 데이터 흐름
- 실행 흐름
- 외부 의존성
- 구조적 제약이나 기술 부채

적지 않는 내용:

- 프로젝트나 서비스의 목적
- 작업 우선순위
- 미결정된 운영 방향
- 현재 작업의 세부 TODO

`ARCHITECTURE`는 프로젝트나 서비스가 기술적으로 어떻게 동작하는지 설명하는 문서다.
작은 서비스에서는 생략할 수 있지만, 모듈이 늘어나거나 데이터 흐름이 복잡해지면 별도로 관리하는 것이 좋다.

---

## 템플릿 사용 방식

각 `*.template.md` 파일은 그대로 수정하지 않고, 실제 프로젝트에 복사해서 사용한다.

예:

- `PROJECT_CONTEXT.template.md` → `PROJECT_CONTEXT.md`
- `CURRENT_TASK.template.md` → `CURRENT_TASK.md`
- `NEXT_TASK.template.md` → `NEXT_TASK.md`

템플릿은 시작점일 뿐이며, 프로젝트나 서비스 상황에 맞게 항목을 줄이거나 추가할 수 있다.
다만 문서의 역할이 바뀔 정도로 섞어 쓰지는 않는다.

---

## 문서 갱신 기준

작업이 끝나면 아래 순서로 문서를 확인한다.

1. `CURRENT_TASK`에 완료된 내용 반영
2. 필요한 결정은 `DECISION_LOG`에 기록
3. 이어서 할 작업은 `NEXT_TASK`에 정리
4. 프로젝트나 서비스의 방향이 바뀌면 `PROJECT_CONTEXT` 갱신
5. 구조가 바뀌면 `ARCHITECTURE` 갱신

모든 문서를 매번 갱신할 필요는 없다.
이번 작업으로 실제로 바뀐 내용이 있는 문서만 갱신한다.
